## 1) Metodi standard ERC-20

Implementazione completa ERC-20 tramite `ERC20Upgradeable` di OZ v5:

- `transfer(address to, uint256 value) → bool`
- `transferFrom(address from, address to, uint256 value) → bool`
- `approve(address spender, uint256 value) → bool`
- `allowance(address owner, address spender) → uint256`
- `balanceOf(address account) → uint256`
- `totalSupply() → uint256`

Tutta la logica di trasferimento converge su `_update(address from, address to, uint256 value)` (pattern OZ v5). Il contratto effettua l'override di `_update` per integrare in ordine: pausa, block check, freeze check, fee.

## 2) Upgrade UUPS + ERC-1822

### Funzionalità

- Upgrade tramite **UUPS Proxy** (`UUPSUpgradeable` + `ERC1967Proxy`)
- Storage slot proxy secondo `ERC-1967`
- Autorizzazione upgrade tramite `_authorizeUpgrade()` protetta da `UPGRADER_ROLE`

### Metodi

> ⚠️ In OZ v5 la funzione `upgradeTo(address)` è stata **rimossa**. Esiste solo:

- `upgradeToAndCall(address newImplementation, bytes memory data)` — se non si deve chiamare nessuna funzione, si passa `""` come `data`

### Inizializzazione

- `initialize(...)` con modifier `initializer` (OZ `Initializable`)
- Nessun costruttore di logica: l'implementazione usa il costruttore **solo** per chiamare `_disableInitializers()`:

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

### `_authorizeUpgrade`

```solidity
function _authorizeUpgrade(address newImplementation)
    internal
    override
    onlyRole(UPGRADER_ROLE)
{}
```

## 3) Access Control

Basato su `AccessControlUpgradeable` di OZ v5.

### Ruoli

| Ruolo | Identificatore | Scopo |
|---|---|---|
| `DEFAULT_ADMIN_ROLE` | `0x00` | **Gestione ruoli** (grant/revoke). Non esegue direttamente azioni operative. |
| `UPGRADER_ROLE` | `keccak256("UPGRADER_ROLE")` | Autorizzare upgrade dell'implementazione |
| `PAUSER_ROLE` | `keccak256("PAUSER_ROLE")` | Sospendere/riprendere i trasferimenti |
| `MINTER_ROLE` | `keccak256("MINTER_ROLE")` | Emettere nuovi token |
| `BURNER_ROLE` | `keccak256("BURNER_ROLE")` | Distruggere token (qualsiasi indirizzo, senza consenso) |
| `FREEZER_ROLE` | `keccak256("FREEZER_ROLE")` | Congelare/scongelare saldi |
| `BLOCKER_ROLE` | `keccak256("BLOCKER_ROLE")` | Bloccare/sbloccare indirizzi |
| `FEE_ADMIN_ROLE` | `keccak256("FEE_ADMIN_ROLE")` | Configurare fee e whitelist |
| `RECOVERER_ROLE` | `keccak256("RECOVERER_ROLE")` | Recuperare token/ETH/NFT bloccati nel contratto |

### Nota architetturale — Opzione B

All'`initialize()`, il deployer riceve **solo** `DEFAULT_ADMIN_ROLE`. Lo script di deploy chiama poi `grantRole(R, X)` per ciascun ruolo operativo verso l'indirizzo configurato, e infine il deployer `renounceRole` sui ruoli che non gli competono. Il `DEFAULT_ADMIN_ROLE` resta al deployer **solo per gestire i ruoli** (grant/revoke), non per eseguire azioni operative.

Su mainnet: il `DEFAULT_ADMIN_ROLE` e `UPGRADER_ROLE` saranno gestiti via **TimelockController + Multisig** (vedi §11).

## 4) Pause (Pausable)

Basato su `ERC20PausableUpgradeable` di OZ v5, che fa override di `_update` con il modifier `whenNotPaused`.

- `pause()` → blocca tutti i trasferimenti (e mint/burn). Protetto da `PAUSER_ROLE`.
- `unpause()` → riattiva i trasferimenti. Protetto da `PAUSER_ROLE`.

### Comportamento in stato di pausa

La pausa **blocca** tutte le operazioni che chiamano `_update`:

✅ **Bloccate:** `transfer`, `transferFrom`, `mint`, `burn`, `transferAndCall`, `transferFromAndCall`, `transferWithAuthorization`, `receiveWithAuthorization`

❌ **Non bloccate** (rimangono operative anche in pausa): `approve`, `permit`, `approveAndCall`, `cancelAuthorization`, `recoverERC20/ETH/ERC721`, `grantRole/revokeRole`, `setFee`, `freeze/unfreeze`, `blockUser/resetUser`, `upgradeToAndCall`, tutte le funzioni `view`

> Vedi la matrice completa in coda al documento.

## 5) Supply Control — Mint + Burn con ruoli separati

### Mint

```solidity
function mint(address to, uint256 amount) external onlyRole(MINTER_ROLE)
```

- Protetto da `MINTER_ROLE`.
- **Nessun cap di supply**: il contratto non impone un massimo di token emettibili.
- Il mint **non paga fee** (esente per costruzione: `from == address(0)` nella logica di `_update`).
- Non è possibile mintare verso un indirizzo bloccato (`blockUser`).

### Burn

```solidity
function burn(address from, uint256 amount) external onlyRole(BURNER_ROLE)
```

- Protetto da `BURNER_ROLE`.
- Il burn **non richiede il consenso del titolare** dell'indirizzo — è un potere di confisca esplicito e intenzionale (pattern stablecoin: freeze → investigazione → burn).
- Il burn **può colpire anche token congelati** (`BURNER_ROLE` sovrascrive il freeze check).
- Il burn **non paga fee** (esente: `to == address(0)`).
- `burnFrom()` è **eliminato**: non esiste nel contratto.

## 6) ERC-1363 — Token con callback + ERC-165

Implementato come modulo custom `ERC20_1363Upgradeable`.

### Metodi

- `transferAndCall(address to, uint256 value) → bool`
- `transferAndCall(address to, uint256 value, bytes memory data) → bool`
- `transferFromAndCall(address from, address to, uint256 value) → bool`
- `transferFromAndCall(address from, address to, uint256 value, bytes memory data) → bool`
- `approveAndCall(address spender, uint256 value) → bool`
- `approveAndCall(address spender, uint256 value, bytes memory data) → bool`

### Callback

- `IERC1363Receiver.onTransferReceived(address operator, address from, uint256 value, bytes data)` — invocata dopo ogni `transferAndCall` / `transferFromAndCall`
- `IERC1363Spender.onApprovalReceived(address owner, uint256 value, bytes data)` — invocata dopo `approveAndCall`

### Nota sulle fee

Quando la fee è > 0 e né `from` né `to` sono in whitelist, la callback `onTransferReceived` riceve il **valore netto** (`value − fee`). È il comportamento corretto: il recipient sa esattamente quanto ha ricevuto.

### ERC-165

`supportsInterface(bytes4 interfaceId)` ritorna `true` per `IERC1363` e `IERC20`.

## 7) EIP-2612 — Permit + EIP-712 + EIP-5267

Implementato tramite `ERC20PermitUpgradeable` di OZ v5 (che include `EIP712Upgradeable`).

### Metodi

- `permit(address owner, address spender, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s)`
- `nonces(address owner) → uint256` — nonce **sequenziale**
- `DOMAIN_SEPARATOR() → bytes32`
- `_hashTypedDataV4(bytes32 structHash) → bytes32`
- `eip712Domain()` (EIP-5267) — ritorna i 7 campi del dominio: `fields, name, version, chainId, verifyingContract, salt, extensions`

### Note

- `permit()` è il meccanismo di **allowance via firma**: non sposta saldi, non è bloccata dalla pausa.
- Il nonce è sequenziale (incrementale): diverso dal nonce arbitrario di EIP-3009.

## 8) EIP-3009 — Transfer With Authorization

Implementato come modulo custom `ERC20EIP3009Upgradeable` (OZ non offre un modulo ufficiale). Usa `EIP712Upgradeable` per la firma (condiviso con il Permit).

### Metodi

- `transferWithAuthorization(address from, address to, uint256 value, uint256 validAfter, uint256 validBefore, bytes32 nonce, uint8 v, bytes32 r, bytes32 s)`
- `receiveWithAuthorization(address from, address to, uint256 value, uint256 validAfter, uint256 validBefore, bytes32 nonce, uint8 v, bytes32 r, bytes32 s)` — solo il `to` può chiamarla
- `cancelAuthorization(address authorizer, bytes32 nonce, uint8 v, bytes32 r, bytes32 s)`
- `authorizationState(address authorizer, bytes32 nonce) → bool`

### Storage (ERC-7201 namespace `advanced.token.eip3009`)

```solidity
struct EIP3009Storage {
    mapping(address authorizer => mapping(bytes32 nonce => bool used)) authorizationState;
}
```

### Nonce

Il nonce EIP-3009 è **arbitrario** (32 byte casuali), diverso dal nonce sequenziale di EIP-2612: i due sistemi usano storage separato e sono incompatibili per design.

### Nota sulle fee

Quando la fee è > 0, il recipient di un `transferWithAuthorization` riceve `value − fee`, non `value`. Poiché `value` era stato firmato off-chain, c'è uno scostamento tra importo firmato e importo ricevuto. Mitigazione: aggiungere gli indirizzi degli integratori alla `feeFreeList`.

## 11) Recupero token dal contratto

Implementato come modulo custom `ERC20RecoverableUpgradeable`. Protetto da **`RECOVERER_ROLE`** (non più `DEFAULT_ADMIN_ROLE`).

### Funzionalità

- Recupero ETH/POL nativo bloccato nel contratto
- Recupero di qualsiasi token ERC-20, incluso il token stesso (`address(this)`)
- Recupero di NFT ERC-721 ricevuti per errore
- **Non bloccata dalla pausa** (funzione di emergenza admin)

### Metodi

```solidity
recoverERC20(address token, address to, uint256 amount)
recoverETH(address payable to, uint256 amount)
recoverERC721(address nft, address to, uint256 tokenId)
```

> ⚠️ I metodi includono il parametro `to` (destinatario) rispetto alla versione precedente che non lo aveva.

### Nota: proxy come fee vault

È ammesso impostare `feeCollector = address(proxy)`. In quel caso le fee si accumulano nel proxy e vengono prelevate tramite `recoverERC20(address(this), treasury, amount)` dal `RECOVERER_ROLE`.

### `receive()` e `IERC721Receiver`

Il contratto implementa:

- `receive() external payable {}` — per ricevere ETH/POL nativo
- `IERC721Receiver.onERC721Received(...)` ritornando il selector — per ricevere NFT via `safeTransferFrom`

## 12) Freeze — Modulo custom `ERC20FreezableUpgradeable`

Reimplementato inline (non da community-contracts). Storage: ERC-7201 namespace `advanced.token.freezable`.

### Attributi (view methods)

| Metodo | Descrizione |
|---|---|
| `frozenOf(address account) → uint256` | Quantità congelata. Se `type(uint256).max` = "frozen all". |
| `availableBalanceOf(address account) → uint256` | `balanceOf − frozenOf`, con clamping a 0. |

### Metodi di scrittura (protetti da `FREEZER_ROLE`)

| Metodo | Effetto |
|---|---|
| `freeze(address account, uint256 amount)` | Imposta `frozen[account] = amount` (overwrite) |
| `freezeAll(address account)` | Imposta `frozen[account] = type(uint256).max` |
| `unfreeze(address account)` | Imposta `frozen[account] = 0` |
| `reduceFrozen(address account, uint256 amount)` | Decrementa `frozen[account]` di `amount` (revert se underflow) |

### Comportamento in `_update`

- Se `from != address(0)` e `frozen[from] != 0`: verifica che `value ≤ balanceOf(from) − frozen[from]`, altrimenti revert con `InsufficientUnfrozenBalance(uint256 requested, uint256 available)`.
- **Mint è esente** (from == address(0)).
- **Burn può colpire token congelati**: `BURNER_ROLE` è esente dal freeze check, per permettere la confisca.

### Separazione poteri

`FREEZER_ROLE` congela → `BURNER_ROLE` brucia. Due ruoli diversi devono agire in sequenza. Il `FREEZER_ROLE` da solo non può prelevare né distruggere token.

## 13) Block — Modulo custom `ERC20RestrictedUpgradeable`

Reimplementato inline (non da community-contracts). Storage: ERC-7201 namespace `advanced.token.restricted`.

### Attributi (view methods)

| Metodo | Descrizione |
|---|---|
| `isBlocked(address account) → bool` | True se l'indirizzo è bloccato |

### Metodi di scrittura (protetti da `BLOCKER_ROLE`)

| Metodo | Effetto |
|---|---|
| `blockUser(address account)` | `blocked[account] = true` |
| `resetUser(address account)` | `blocked[account] = false` |

### Comportamento in `_update`

- Se `blocked[from] == true` **oppure** `blocked[to] == true` → revert con `AddressBlocked()`.
- **Vale anche per mint e burn**: non si può mintare verso un indirizzo bloccato, né bruciare da uno bloccato. Questo impedisce qualsiasi aggiramento del blocco.

## 14) Fee moderna con whitelist e cap

Implementata come modulo custom `ERC20FeeUpgradeable`. Storage: ERC-7201 namespace `advanced.token.fee`.

### Storage

```solidity
struct FeeStorage {
    uint256 fee;                          // basis points (0–999)
    address feeCollector;
    mapping(address => bool) isFeeFree;
}
```

### Costanti

```solidity
uint16 public constant MAX_FEE_BASIS_POINTS = 999; // 9,99% massimo assoluto
```

### Metodi pubblici (protetti da `FEE_ADMIN_ROLE`)

| Metodo | Descrizione |
|---|---|
| `setFee(uint256 newFee)` | Imposta la fee in bp. Revert se `newFee > 999`. |
| `setFeeCollector(address collector)` | Imposta il destinatario delle fee. |
| `addFeeFree(address account)` | Aggiunge un indirizzo alla whitelist. |
| `removeFeeFree(address account)` | Rimuove un indirizzo dalla whitelist. |

### Metodi view (pubblici)

| Metodo | Descrizione |
|---|---|
| `fee() → uint256` | Fee corrente in bp |
| `feeCollector() → address` | Indirizzo fee collector |
| `isFeeFree(address account) → bool` | True se l'indirizzo è in whitelist |

### Logica in `_update` — quando si applica la fee

La fee è applicata **solo se tutte** queste condizioni sono vere simultaneamente:

1. `fee != 0` (short-circuit: se zero, nessun calcolo)
2. `from != address(0)` (non è un mint)
3. `to != address(0)` (non è un burn)
4. `!isFeeFree[from]` (sender non in whitelist)
5. `!isFeeFree[to]` (recipient non in whitelist)

Se applicata: `feeAmount = (value × fee) / 10.000`. Due `Transfer` distinti: uno verso `feeCollector`, uno verso `to`.

### Interazione con EIP-3009 e ERC-1363

Quando la fee è attiva e le parti non sono in whitelist:

- `transferWithAuthorization` / `receiveWithAuthorization`: il recipient riceve `value − fee`, nonostante `value` fosse stato firmato off-chain. **Mitigazione:** aggiungere alla `feeFreeList` gli integratori che necessitano importi esatti.
- `transferAndCall` / `transferFromAndCall` (ERC-1363): la callback `onTransferReceived` riceve il **valore netto** (`value − fee`).

## 15) Storage Layout — ERC-7201 Namespaced Storage

Tutti i moduli custom usano il pattern **ERC-7201** per lo storage, coerentemente con OZ v5.

**Niente `__gap`**: con namespaced storage ogni modulo ha uno slot indipendente derivato da hash — non ci sono collisioni tra moduli ereditati, e non servono array gap come nel vecchio pattern OZ v4.

### Namespace assegnati

| Modulo | Namespace ERC-7201 |
|---|---|
| Fee | `advanced.token.fee` |
| Freezable | `advanced.token.freezable` |
| Restricted | `advanced.token.restricted` |
| EIP-3009 | `advanced.token.eip3009` |

Aggiungere campi a una struct è sicuro purché si aggiungano **in fondo** e non si riordini/rinomini quelli esistenti. Il plugin `@openzeppelin/hardhat-upgrades` verifica automaticamente la compatibilità a ogni upgrade.

## 16) Ordine canonico dei controlli in `_update`

```
1. PAUSE      → modifier whenNotPaused (da ERC20PausableUpgradeable)
2. BLOCK      → revert AddressBlocked() se from o to bloccati
3. FREEZE     → revert InsufficientUnfrozenBalance se from non ha abbastanza saldo libero
4. FEE        → calcolo e split (solo se fee > 0, non mint, non burn, non whitelist)
5. SETTLEMENT → chiamata a super._update(from, to, value)
```

## 17) Configurazione iniziale — `initialize`

```solidity
function initialize(
    string memory name_,
    string memory symbol_,
    uint256 initialSupply_,
    address initialHolder_,
    uint256 initialFee_,
    address feeCollector_,
    address defaultAdmin_
) public initializer
```

**Comportamento:**

1. Inizializza tutti i moduli OZ (`__ERC20_init`, `__ERC20Permit_init`, `__ERC20Pausable_init`, `__AccessControl_init`, `__UUPSUpgradeable_init`) e i moduli custom.
2. Assegna `DEFAULT_ADMIN_ROLE` a `defaultAdmin_`.
3. Imposta `fee = initialFee_` con check `≤ MAX_FEE_BASIS_POINTS`.
4. Imposta `feeCollector = feeCollector_` (revert se `address(0)`).
5. Se `initialSupply_ > 0`, esegue `_mint(initialHolder_, initialSupply_)` — **chiamata diretta alla funzione interna**, senza richiedere `MINTER_ROLE`.
6. Nessun altro ruolo viene assegnato: lo script di deploy chiama `grantRole` separatamente.

**`decimals()`** ritorna `18` (default OZ, nessun override).

**Valori di default per testnet:**

- Supply iniziale: 10.000 token (`10_000 × 1e18`)
- Holder iniziale: indirizzo del deployer
- Fee iniziale: 10 bp (0,10%)
